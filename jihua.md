

# 项目初步目标说明

## 项目名称（暂定）

**终端命令历史增强记录工具**

---

## 一、项目背景

在日常开发、调试、环境配置和问题排查过程中，终端命令行是非常核心的工作入口。无论是安装依赖、启动服务、执行脚本、排查报错，还是进行 Git 操作，很多关键步骤都发生在终端中。

虽然 Bash、Zsh 等 Shell 本身提供了 `history` 功能，可以查看历史命令，但原生历史记录仍然存在一些明显不足：

1. **历史记录只保存命令文本，缺少上下文信息**
   普通的 history 通常只能看到执行过什么命令，但看不到这条命令是什么时候执行的、当时所在目录是什么、执行是否成功、是否属于某次特定任务。

2. **命令记录不够结构化，后续检索和分析能力弱**
   原生命令历史更像简单文本列表，不适合做标签管理、关键词检索、按项目分类、错误命令统计等进一步处理。

3. **很多重要操作难以沉淀为可复盘的执行日志**
   例如某次环境配置、一次调试流程、一次部署尝试，用户可能连续执行了多条命令，但事后很难把这些关键步骤提取出来形成清晰记录。

4. **对个人学习、排障和经验复用帮助有限**
   对学生和开发者来说，很多终端操作其实是高频重复的。如果能把关键命令及其场景保存下来，后面就能更快复现操作流程、回顾问题处理过程、积累个人命令知识库。

基于以上问题，本项目希望实现一个轻量的终端工具：
用户在执行完命令后，可以主动调用一个自定义命令，将上一条或最近几条命令提取出来，并以结构化方式保存为日志，便于后续查询、整理和复盘。

---

## 二、项目目标

本项目的目标是设计并实现一个 **面向终端用户的命令记录增强工具**，在不替代现有 Shell 的前提下，为命令历史增加“结构化记录、上下文补充、日志存储和后续检索”能力。

具体来说，项目希望解决以下问题：

### 1. 让关键命令的记录更加可控

用户不一定希望把所有命令都自动记录下来，而是希望在完成某一步操作后，主动决定是否保存这次命令。
因此，本工具采用“手动触发记录”的模式：用户在执行完某条命令后，可通过调用工具命令，将上一次或最近几次命令记录下来。

### 2. 让命令记录不只是“文字历史”，而是“带上下文的操作日志”

除了命令本身，还希望记录一些有价值的信息，例如：

* 执行时间
* 所在目录
* Shell 类型
* 主机名/用户名
* 命令执行状态
* 用户添加的标签或备注

这样记录下来的内容更接近“操作日志”，而不只是简单的终端历史。

### 3. 为后续查询、复盘和经验积累提供基础

保存后的命令记录应支持后续按关键词、标签、目录、时间等维度查找。
例如：

* 查找某次 Git 操作记录
* 回顾某次环境搭建过程
* 查看最近失败过的命令
* 统计某类命令的使用情况

### 4. 提升工具的实用性和工程感

该项目不仅是一个简单脚本，而是希望做成一个相对完整的小型 CLI 工具，具备清晰的命令设计、结构化存储方式、基础配置能力和一定的可扩展性，为后续增加更多功能打下基础。

## 三、核心使用场景

### 场景 1：记录刚执行完的一条关键命令

用户刚刚执行了一条重要命令，例如：

```bash
git rebase main
```

执行后，用户希望把这条命令记下来，于是输入：

```bash
clog
```

工具会自动提取上一条命令，并记录为一条日志，附带时间、目录等信息。

---

### 场景 2：记录最近一组连续操作

用户刚完成一段配置流程，例如最近 3 条命令分别是：

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

这时用户希望把这组步骤一起记录下来，于是输入：

```bash
clog -n 3
```

工具会提取最近 3 条命令并保存，形成一个小型操作片段，便于后续复现。

---

### 场景 3：给命令记录打标签或备注

用户在排查某个 bug 时执行了一系列命令，希望后面还能快速找到，于是输入：

```bash
clog -n 5 -t debug-auth -m "排查登录接口 401 问题"
```

工具会把最近 5 条命令记录下来，并打上标签和备注。

---

### 场景 4：查看已保存的命令日志

用户之后希望查看自己记录过的内容，可以执行：

```bash
clog list
```

或者：

```bash
clog search git
```

这样就能快速定位过去保存过的关键命令片段。

---

## 四、项目定位

这个项目的定位不是“替代 Shell 的历史系统”，而是：

## **对原生命令历史进行增强和结构化沉淀的辅助工具**

也就是说，它更像一个建立在 Bash/Zsh 历史机制之上的“二次记录层”。

它强调的是：

* 用户主动选择要记录哪些命令
* 对关键命令附加上下文信息
* 把命令从“历史列表”变成“可检索的日志条目”
* 为个人学习、调试、复盘和经验积累服务

---

## 五、第一版功能目标

第一版不追求做得很大，而是先实现一个 **完整、可用、可展示** 的最小版本。

### 第一版计划支持的能力

#### 1. 记录上一条命令

用户执行 `clog` 后，自动获取上一条命令并保存。

#### 2. 记录最近 N 条命令

用户可以通过参数指定记录最近几条命令，例如 `clog -n 3`。

#### 3. 记录上下文信息

每条记录至少保存以下字段：

* 时间戳
* 命令内容
* 当前工作目录
* 用户名
* 主机名
* Shell 类型
* 执行状态码（若能稳定获取）

#### 4. 支持标签和备注

用户可以为记录添加：

* tag
* note/message

#### 5. 结构化存储

第一版可采用 JSON Lines 方式保存，保证简单易实现，同时便于后续迁移到 SQLite。

#### 6. 查看和查询记录

提供基础命令：

* 查看记录列表
* 关键词搜索
* 按标签过滤

#### 7. 基础敏感信息处理

对明显可能包含敏感内容的命令做初步处理，例如：

* token
* password
* secret
* key

第一版可以先做简单脱敏，而不是复杂的安全系统。

---

## 六、第一版暂不处理的内容

为了控制项目规模，以下内容可以明确列为“后续扩展”，第一版先不做：

* 所有 Shell 的完全兼容
* 自动后台无感记录全部命令
* 复杂权限管理
* 云端同步
* 多设备共享
* GUI 图形界面
* 高级统计分析
* AI 自动总结命令会话
* 与 IDE、Git 平台或远程服务器深度集成

这样可以避免项目一开始就失控。

## 八、技术实现目标（初步）

从实现思路上，项目计划采用：

* **Shell function + Python 程序** 的组合方式
  由 shell function 在当前 shell 环境中获取最近历史命令，再交给 Python 程序完成参数处理、数据清洗和日志存储。

* **Python 作为核心实现语言**
  负责：

  * 参数解析
  * 数据结构定义
  * 日志格式化
  * JSON Lines 存储
  * 查询命令实现

* **优先支持 Bash，后续再考虑 Zsh**
  第一版先把单一环境做好，减少兼容性负担。

---

## 九、项目完成后的展示方式

这个项目完成后，可以作为一个“工具型项目”写入简历，重点突出：

* 面向真实终端使用场景设计 CLI 工具
* 支持历史命令提取、结构化存储与查询
* 通过 Shell 与 Python 结合解决命令历史获取问题
* 完成日志字段设计、敏感信息处理和基础检索功能

它不是大型平台项目，但如果完成度高、文档清晰、使用体验顺畅，会是一个很有个人特点的小项目。

## 十、一句话总结

**本项目旨在将终端中的临时命令历史，转化为可主动保存、可附加上下文、可长期检索的结构化操作日志，帮助用户更好地复盘、管理和复用命令行工作过程。**
